# Эссе по вопросам экзамена «Микросервисная архитектура 2026»

## Введение

Микросервисная архитектура, контейнеризация и оркестрация уже стали стандартом разработки распределённых приложений, особенно в веб-среде и менеджменте крупных сервисов. В этом эссе я рассмотрю ключевые технологии и практики и покажу, как они взаимосвязаны на примере реального проекта.

---

## 1. Микросервисная архитектура

### 1.1 Разделение монолита на микросервисы

Главная цель микросервисного подхода - декомпозиция приложения на автономные сервисы, каждый из которых отвечает за одну бизнес‑функцию. Каждый сервис имеет собственное окружение при выполнении, что исключает их жёсткую связность и созависимость внутри системы.

Преимущества: независимое развёртывание, масштабирование по нагрузке, возможность использовать разные стеки технологий, отказоустойчивость (микросервисы падают по одному, не прерывая работу всей системы).

Платой становятся сложности с распределёнными транзакциями и сетевой задержкой.

### 1.2 Передача данных между микросервисами

Синхронное взаимодействие реализуется в основном через REST API (HTTP/JSON) или более технологичный, производительный, но более сложный в интеграции gRPC за счёт структуры генерации кода на основе .proto файлов (построен на основе HTTP\2, поддерживает потоковую передачу в обе стороны и использует protobuf для сериализации сообщений). Асинхронная передача выполняется через очереди сообщений (например, RabbitMQ, Kafka) или события через WebHook.

### 1.3 Stateless и Stateful приложения

Статистические (stateless) сервисы не хранят состояние между запросами и легко масштабируются горизонтально. Пример: API Gateway, сервис обработки изображений. Состояние (stateful) хранится во внешних системах: базах данных, кэшах, объектных хранилищах. Это позволяет любому экземпляру сервиса работать с одними и теми же данными.

В моём проекте все основные микросервисы - stateless, состояния вынесены в Postgres и Minio. Это упрощает оркестрацию в Kubernetes и позволяет легко перезапускать поды без потери данных.

### 1.4 Передача параметров конфигурации

Конфигурация должна быть отделена от кода. Для этого используются переменные окружения, файлы конфигурации (.yaml, .json, .cfg, .env и т.д., ну или просто данные в строках TXT) или специализированные системы (Consul, etcd). В Kubernetes я применяю ConfigMap и Secret для хранения параметров и чувствительных данных.

В моём проекте все конфигурационные настройки (хост, порт, учётные данные) передаются через переменные окружения, что позволяет изменять поведение без пересборки образа.

### 1.5 Логирование работы приложения

Распределённые системы требуют централизованного сбора и анализа логов. Для лучшей читаемости логов обычно используются уровни от ERROR до DEBUG, различные поля логов (timestamp, зарегистрированный пользователь, сервис, идентификатор запроса, трассировка вызовов и т.д.). Для поддержания эфимерности системы логи пишутся в stdout и stderr, а оркестратор может собирать их в файлы для анализа, ведь так удобнее.


Я использую структурированное логирование в формате JSON (через `log/slog`), что упрощает парсинг и интеграцию с ELK или Loki. Каждый сервис пишет логи в stdout, а оркестратор собирает их.

В проекте я логирую каждый ключевой этап: получение запроса, генерацию URL, начало/завершение обработки, отправку уведомления, ошибки. Логи содержат идентификатор запроса, что позволяет трассировать цепочку вызовов.

### 1.6 Использование метрик для мониторинга

Метрики - это числовые показатели состояния системы. Я внедряю библиотеку Prometheus client, которая предоставляет эндпоинт `/metrics` с настраиваемыми счётчиками, гистограммами и датчиками. В проекте я добавил счётчик HTTP‑запросов с разбивкой по методу, пути и статусу ответа.

Метрики собираются Prometheus, визуализируются в Grafana, позволяют строить дашборды и настраивать алерты. Это даёт оперативную информацию о нагрузке, ошибках и производительности.

Виды метриков тоже бывают разные - от обычных счётчиков до процентилей и гистограмм.

## 2. Технология виртуализации Docker

### 2.1 Основные особенности

Docker - это система контейнеризации, использующая изоляцию на уровне ОС (namespaces, cgroups). Контейнеры легче виртуальных машин, т.к. разделяют ядро хоста, запускаются за секунды и потребляют меньше ресурсов. Образы строятся из слоёв, что позволяет переиспользовать и кэшировать слои, ускоряя сборку и уменьшая объём передаваемых данных.

Docker предоставляет возможность создать заведомо рабочие контейнеры с определённой логикой - они не зависит от установленных на машине библиотек - внутри контейнера может выполняться любой код, если есть необходимые зависимости.

### 2.2 Сборка для интерпретируемых и компилируемых языков

Для интерпретируемых языков (Python, Node.js, Ruby) образ обычно содержит интерпретатор, код приложения и зависимости. Сборка проста: копируем код, устанавливаем пакеты через менеджер (pip, npm). Многослойность позволяет кэшировать установку зависимостей, если не менялся файл требований.

Для компилируемых языков (Go, Java, C++) применяется multi‑stage build. Сначала в образе с компилятором и утилитами собирается бинарный файл, затем он копируется в минимальный финальный образ (например, на основе Alpine). Это даёт малый размер образа, что критично для скорости загрузки и развёртывания.

В моём проекте для Go я использую двухэтапную сборку: сначала `golang:alpine` с компиляцией, затем `alpine` с готовым бинарником.

### 2.3 Docker Compose

Это инструмент для локального запуска многоконтейнерных приложений. Он описывает все сервисы, сети, тома в одном YAML‑файле. Основные возможности: определение зависимостей между сервисами, проброс портов, управление переменными окружения, автоматическое создание сетей.

Compose идеально подходит для разработки и тестирования, т.к. позволяет одним `docker-compose up` поднять весь стек. Я применял его, чтобы быстро проверить интеграцию сервисов без развёртывания в Kubernetes.

### 2.4 Использование Docker на этапе разработки

Docker обеспечивает единообразие окружения разработки, тестирования и продакшена. Разработчик запускает контейнеры локально с теми же версиями ПО, что и на сервере. Это устраняет проблему «у меня работает, а на сервере нет». Также Docker ускоряет обратную связь: можно пересобрать и перезапустить один сервис, не пересобирая всё приложение.

### 2.5 Примеры контейнеров

- Базы данных: Postgres, MySQL, Redis.
- Брокеры сообщений: RabbitMQ, Kafka.
- Кэширующие серверы: Memcached.
- Серверы объектов: Minio.
- Прокси и балансировщики: Nginx, Traefik.
- Языковые окружения: Python, Node.js, Go, Java.

### 2.6 Основные команды Docker

Предоставлю самые основные часто используемые команды, которые сам применял во время разработки и тестирования.
- `docker build -t myimage .` - сборка образа; `-f` указать Dockerfile, `--no-cache` отключить кэш, `--build-arg` передать аргументы.
- `docker run -d -p 8080:80 myimage` - запуск контейнера в фоне (`-d`); `--name` задать имя, `-v` монтировать том, `-e` установить переменную окружения, `--rm` автоматически удалить после остановки, `--restart` политику перезапуска.
- `docker ps` - список запущенных контейнеров; `-a` показать все (включая остановленные), `-q` только ID.
- `docker logs <container>` - логи контейнера; `-f` следить в реальном времени, `--tail N` показать последние N строк.
- `docker exec -it <container> sh` - выполнить команду в запущенном контейнере; `-i` интерактивный режим, `-t` псевдо-TTY (обычно используются вместе), `-e` передать переменную окружения.
- `docker stop/rm <container>` - остановить (`stop`) или удалить (`rm`) контейнер; `-f` принудительно (force).
- `docker system prune` - очистка неиспользуемых ресурсов (образы, контейнеры, сети, build cache); `-a` удалить все неиспользуемые образы, `-f` без подтверждения.
- `docker-compose up/down` - запуск/остановка стека; `up -d` в фоновом режиме, `--build` пересобрать образы, `down -v` удалить тома.

## 3. Технология Kubernetes

### 3.1 Основные особенности

Kubernetes - это платформа оркестрации контейнеров, обеспечивающая автоматическое развёртывание, масштабирование и управление контейнеризованными приложениями. Она абстрагирует инфраструктуру, позволяя запускать приложения в любом облаке или локальном центре обработки данных. Ключевые идеи: декларативность, самовосстановление, горизонтальное масштабирование, автоматические обновления.

### 3.2 Компоненты системы

- **Pod** - минимальная вычислительная единица, один или несколько контейнеров, разделяющих сеть и тома.
- **ReplicaSet** - гарантирует, что заданное число подов работает в любой момент.
- **Deployment** - декларативный способ управления ReplicaSet и обновлениями (rolling update, откат).
- **Service** - абстракция, предоставляющая стабильный сетевой доступ к группе подов (ClusterIP, NodePort, LoadBalancer).
- **Ingress** - внешний HTTP/HTTPS‑маршрутизатор, позволяет публиковать сервисы за пределами кластера, поддерживает TLS и правила маршрутизации по хостам/путям.
- **ConfigMap** - хранение несекретных конфигурационных данных в виде пар ключ-значение.
- **Secret** - аналогичен ConfigMap, но для чувствительных данных (пароли, токены), хранится в base64, может быть зашифрован.
- **NetworkPolicy** - управление трафиком между подами.
- **PodDisruptionBudget** - обеспечение доступности при обновлениях.
- **HorizontalPodAutoscaler** - автоматическое масштабирование пода в зависимости от загрузки и работы.


Также есть PersistentVolume/PersistentVolumeClaim - управление хранилищем, StatefulSet - для stateful‑приложений, DaemonSet - запуск пода на каждой ноде, Job/CronJob - одноразовые и периодические задачи.

### 3.3 Особенности маршрутизации в кластере

Внутри кластера работает внутренний DNS (CoreDNS), который резолвит имена сервисов в ClusterIP. Он может балансировать трафик между подами по нескольким стратегиям (round‑robin по умолчанию). Ingress‑контроллер (например, nginx‑ingress) обрабатывает входящие HTTP‑запросы, направляя их на нужные контейнеры микросервисов на основе правил, что обеспечивает внешнюю маршрутизацию без необходимости выделять NodePort для каждого сервиса.

Для межсервисного общения я использую внутренние имена (например, `http://img-service:8081`), которые резолвятся в ClusterIP.

### 3.4 Передача параметров конфигурации

В Kubernetes я передаю параметры через ConfigMap и Secret. Они могут быть подключены как переменные окружения (`envFrom`) или смонтированы как файлы. Это отделяет конфигурацию от образа, позволяя изменять параметры без пересборки.

Пример: в проекте я храню пароли БД в Secret, а переменные по типу `S3_EXTERNAL_URL` - в ConfigMap (или напрямую в Deployment), чтобы можно было обновлять без пересборки образа. При изменении конфигурации я просто обновляю ресурсы и перезапускаю поды.

### 3.5 Управление работой приложений

Deployment-манифест (спецификация) обеспечивает декларативные обновления: я задаю желаемое состояние, а Kubernetes постепенно заменяет старые поды новыми (rolling update). При возникновении ошибок можно откатиться к предыдущей версии. Liveness и readiness пробы позволяют контролировать здоровье подов: Kubernetes перезапускает зависшие поды и не направляет трафик на неподготовленные.

### 3.6 Примеры манифестов

Ниже приведены упрощённые фрагменты манифестов, которые я применял в проекте.

**Deployment** для API‑сервиса:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
  namespace: sirius
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api-service
  template:
    metadata:
      labels:
        app: api-service
    spec:
      containers:
      - name: api-service
        image: api-service:latest
        ports:
        - containerPort: 8080
        env:
        - name: DB_HOST
          value: "postgres"
        - name: JWKS_URL
          value: "https://id.klsh.ru/..."
        readinessProbe:
          httpGet:
            path: /healthz
            port: 8080
        livenessProbe:
          httpGet:
            path: /healthz
            port: 8080
```

**Service** для доступа внутри кластера:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-service
  namespace: sirius
spec:
  selector:
    app: api-service
  ports:
  - port: 8080
    targetPort: 8080
```

**Ingress** для внешнего доступа:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-service
  namespace: sirius
spec:
  ingressClassName: nginx
  rules:
  - host: api.sirius.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
```

**ConfigMap** для init‑скрипта БД:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: postgres-init
  namespace: sirius
data:
  init.sql: |
    CREATE TABLE IF NOT EXISTS public.image (...)
```

**Job** для однократной инициализации Minio:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: minio-init
  namespace: sirius
spec:
  template:
    spec:
      containers:
      - name: mc
        image: minio/mc:latest
        command: ["sh", "-c", "mc alias set local http://minio:9000 s3user s3pass && mc mb ..."]
      restartPolicy: OnFailure
```

## Заключение

Подведя итог - микросервис - это рабочая единица. Докер позволяет упаковать этот микросервис со всем требуемым ПО в образ, который можно развернуть на любой системе. Кластер Kubernetes позволяет организовывать эти контейнеры и управлять их жизненным циклом, он имеет сложную внутреннюю архитектуру и обладает функциями балансировщика трафика, автоматического менеджера контейнеров и т.д.
